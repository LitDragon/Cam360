# Cam360技术架构文档

- 文档版本：v5.0
- 更新时间：2026-04-23
- 文档状态：当前约束版
- 适用范围：Cam360 当前仓库

## 1. 目标

本文件只维护架构边界、路由契约、依赖注入和阶段性演进路线。当前代码事实和技术基线以根目录 `PROJECT_CONTEXT.md` 为准。

## 2. 根结构

- 生命周期入口使用 UIKit 桥接，SwiftUI 负责根视图渲染。
- `AppBootstrap` 负责解析启动参数、初始化仓储与偏好存储，并计算初始路由。
- `AppRouter` 只负责根级分流和主 tab 选择，不承担 Feature 内部业务状态。
- `AppContainer` 统一组合仓储和 Feature Store，再把依赖下发给根视图。
- `AppRootView` 只根据根路由切换 onboarding 和主界面，不在这里扩散业务判断。

## 3. 模块边界

- `App` 目录只放生命周期、根路由、依赖装配和根导航。
- `Features` 目录承载页面、用户动作和页面级状态；默认保留 `View + Store + Route + Components(可选)`。
- `Core` 目录只放跨 Feature 复用的稳定薄能力，不引入重型 DI 或分层框架。
- `Storage` 和 `Device` 这类长生命周期或跨功能依赖，统一从 `AppContainer` 进入 Feature。

## 4. 路由与依赖注入

- 根级路由只区分 `onboarding` 和 `main(MainTab)`。
- onboarding 是否完成由 `AppPreferenceStore` 决定，当前启动分流逻辑集中在 `AppBootstrap`。
- 当前 `AppContainer` 已组合：
  - `KnownDeviceRepository`
  - `AppPreferenceStore`
  - `DeviceOnboardingStore`
  - `DeviceListStore`
  - `LivePreviewStore`
  - `PlaybackStore`
  - `DownloadsStore`
  - `SettingsStore`
- Feature 内部新增依赖时，优先继续沿用容器装配，不把依赖解析散落到 View 层。

## 5. 会话与 Feature 责任

- `DeviceSession` 的目标是统一承接连接、握手、忙碌、恢复、失败和断开语义。
- `LivePreview`、`Playback`、`Downloads`、`Settings` 这类 Feature 只消费会话和能力状态，不直接持有底层连接、播放器或下载任务控制权。
- 新增真实设备能力时，先收敛到 `DeviceSession` 或稳定仓储接口，再接到具体 Feature。

## 6. M1+ 路线
按以下顺序逐步接入真实能力：

1. AP onboarding 真实接入
2. `DeviceSession` 作为统一状态源
3. 实时预览
4. 回放
5. 下载导出
6. 真实设备设置读写
