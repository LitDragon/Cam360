# Cam360 AI 操作手册

本文件是后续 AI 在本仓库工作的第一入口，用于约束每次改动的方向、边界和验证口径。

## 1. 使用方式

- 新会话默认先读本文件。
- 如果任务涉及架构、跨模块改动、路由、状态归属、存储、测试策略，继续读 [AGENTS.md](AGENTS.md) 和 [Cam360技术架构文档.md](Cam360技术架构文档.md)。
- 如果只是单文件 UI 文案、小样式、小 bugfix，通常读完本文件即可开始。

## 2. 当前项目状态

- 当前阶段是 `M0 已基本完成，M1+ 待接入`。
- 已落地的是：
  - UIKit 生命周期桥接 + SwiftUI 根视图
  - `AppBootstrap` / `AppRouter` / `AppContainer`
  - 主界面 4-tab 骨架
  - 最小 DesignSystem
  - `UserDefaults` 驱动的已知设备与偏好存储
  - 最小单元测试与 UI 冒烟测试
- 尚未落地真实链路：
  - AP onboarding
  - `DeviceSession`
  - 实时预览
  - 回放
  - 下载导出
  - 真实设备设置读写

重要：不要把文档中的 M1+ 目标态当成仓库现状。

## 3. 版本与平台策略

- 最低兼容：`iOS 13`
- 主路径实现：优先采用 `iOS 15` 时代稳定、可读、维护成本低的写法
- 增强能力：`iOS 17+` 特性必须放在 `#available(iOS 17, *)` 分支内
- 禁止让 `iOS 17+` 能力影响主路径的启动、路由、测试和基础 UI

## 6. 当前实现事实

- 当前状态模型以 `ObservableObject`、`@Published`、`@ObservedObject`、`@State` 为主。
- 当前本地持久化以 `UserDefaults` 为主，不默认引入 `SwiftData`。
- 当前主导航由 `AppBootstrap` + `AppRouter` + `MainTabView` / `MainTabBar` 驱动。
- 当前 UI 重点是页面骨架、路由入口、最小设计系统，不是设备协议与媒体链路。

## 7. UI 与交互规范

### 7.1 项目内规则

- 优先使用原生 `SwiftUI` 能力，不默认引入第三方整套 UI 库。
- 导航容器、弹窗、sheet、进度指示优先使用原生组件。
- 自研组件只封装样式和少量交互约束，不封装业务逻辑。
- 设计系统保持最小集合：语义色、字体层级、间距、圆角、阴影、图标尺寸、基础反馈组件。
- 不要在项目早期扩张成大而全组件库。

### 7.2 Apple 平台设计规范

以下规则基于 Apple Human Interface Guidelines，适合作为本项目的默认 UI 判断标准：

- 优先使用系统熟悉的导航与控件模式，让界面“像 iOS”，不要为了差异化破坏平台习惯。
- 尊重 safe area、系统边距和系统栏位，不要让主要内容、交互元素贴边。
- 避免无必要的全宽按钮；按钮默认应和系统边距对齐。
- 使用系统文字层级和 Dynamic Type 友好的排版；不要写死只适合默认字号的布局。
- 文字、图标和布局必须在大字号、深色模式、横竖屏和更长文案下保持可读。
- 优先使用 `SF Symbols` 作为系统语义图标；如果图标承载信息，必须有可访问性标签。
- 不要把 `SF Symbols` 当品牌 logo 或商标替代物使用。
- 顶层区域切换用 Tab，当前页面动作放导航栏 / toolbar，不要混淆导航和动作语义。
- 主要任务突出，次要信息可延后显露；不要让页面一上来堆满控制项。
- 交互控件默认满足至少 `44x44pt` 可点击区域。
- 不依赖单一颜色表达状态；要同时给出文案、图标或层级差异。
- 保持足够对比度，避免过细字体、低对比度文本、难点中的次级按钮。
- 状态栏默认保持可见；只有沉浸式媒体场景才考虑弱化或隐藏。
- 涉及实时预览、回放等媒体页面时，应优先保证主内容可见，不让导航和装饰抢占画面。
- 有权限、失败、重试时，优先给明确的下一步动作，不要只给“失败了”。

### 7.3 可访问性底线

- 为按钮、图标、图片、状态视图提供清晰的 accessibility label/value/hint。
- 支持 VoiceOver 基本阅读顺序，不制造无法聚焦的关键操作。
- 减少纯动画传达关键信息；状态变化要有可读文案。
- 不做依赖颜色区分的唯一反馈。

## 8. 测试与验证口径

- 默认先做最窄验证：最小编译、最小单测、最小 UI 冒烟。
- 涉及启动分流、主 Tab、页面骨架时，补对应 UI 冒烟测试。
- UI 截图回归测试默认使用 `SnapshotTesting`，放在 `Cam360Tests/`，不要引入到 app target。
- 当前优先覆盖：
  - `AppBootstrap` 启动分流
  - `AppRouter` 路由切换
  - 本地 Repository / Store 读写
- 不声明成功，除非实际编译或测试已通过。

当前最小 UI 冒烟路径：

1. 首次进入 App，显示 `DeviceOnboarding`
2. 强制进入主界面并完成四个主 Tab 切换

## 9. 何时必须同步文档

如果改动涉及以下内容，除了代码外，也要同步更新相关文档：

- 目录结构或 Feature 边界变化
- 启动分流、路由、主 Tab 信息架构变化
- 存储策略变化
- 测试口径变化
- M0 / M1+ 的阶段定义变化
- UI 基础规范或 DesignSystem 的新硬约束

同步顺序：

1. 更新本 `README.md`
2. 如涉及深层架构事实，再更新 `AGENTS.md`
3. 如涉及阶段目标、模块职责、目标态设计，再更新架构文档

## 10. AI 工作清单

开始前：

- 先确认任务属于 UI 骨架、现有代码修正，还是 M1+ 真实能力接入。
- 先确认改动是否会越过 `App` / `Feature` / `Core` 边界。
- 先确认是否需要补测试或同步文档。

实现时：

- 优先最小改动。
- 优先沿用现有命名、现有模式、现有依赖流向。
- 不把未来架构直接提前“补全”到当前阶段。
- 不在没有证据的情况下假设真实设备链路已存在。

结束前：

- 说明改了什么。
- 说明实际做了什么验证。
